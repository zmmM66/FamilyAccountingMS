## Auth And Backend Behavior

- User/family IDs are case-sensitive.
- Custom ID rule: only letters and numbers, length <= 10.
- Password rule: at least 8 chars and must contain both letters and numbers.
- Temporary passwords for member creation and resets are randomly generated by the backend.
- Registration/join-family require one-time math captcha from `GET /api/captcha`.
- Registering a family directly logs the creator in by returning `token`, `user`, and scoped `state`.
- Login uses user ID + password; pending/rejected users cannot log in.
- Join-family creates a pending user until creator/admin approval.
- Member approval/rejection API is only for pending join requests; active members should be removed through `DELETE /api/members/{id}` instead of changing status.
- Creator registration only allows adult family roles; backend also forces creator `adult_type = "adult"`.
- Password reset flow:
  - public user submits `POST /api/password-resets`
  - duplicate pending reset requests for the same user are reused instead of creating another row
  - creator/admin handles `PATCH /api/password-resets/{id}`
- Backend auth token is HMAC-signed with subject, expiry, and user token version; secret comes from `FAMILY_LEDGER_TOKEN_SECRET` or persists in `data/.token_secret`.
- System admin is a separate auth subject, not a family user:
  - default login ID: `ADMIN_`
  - default password: `ADMIN_PASSWORD`
  - env overrides: `FAMILY_LEDGER_ADMIN_ID`, `FAMILY_LEDGER_ADMIN_PASSWORD`
  - the default credentials are enabled for this local homework app; custom env passwords still need to satisfy the normal strong-password rule
  - auth tokens include an expiry and user token version; password resets invalidate old user tokens
  - token subject uses `system-admin:<admin-id>`
  - ordinary `/api/state` rejects system-admin tokens; `/api/admin/*` rejects ordinary user tokens.
- Frontend does not invent a local default password for backend-managed users; offline/local fallback login only works for users that actually have a locally stored prototype password.
- Important backend routes in `backend/server.py`:
  - `GET /api/health`
  - `GET /api/captcha`
  - `GET /api/state`
  - `POST /api/auth/login`
  - `POST /api/admin/login`
  - `GET /api/admin/state`
  - `POST /api/families`
  - `POST /api/families/join`
  - `DELETE /api/families/{id}`
  - `POST /api/password-resets`
  - `PATCH /api/password-resets/{id}`
  - `PATCH /api/members/{id}/status`
  - `PATCH /api/members/{id}/role`
  - `PATCH /api/members/{id}/transfer-creator`
  - `DELETE /api/members/{id}`
  - `PATCH /api/admin/families/{id}`
  - `PATCH /api/admin/families/{id}/creator`
  - `DELETE /api/admin/families/{id}`
  - `PATCH /api/admin/users/{id}`
  - `PATCH /api/admin/users/{id}/password`
  - `PATCH /api/admin/password-resets/{id}`
- Backend protection currently includes:
  - only creator/admin are managers
  - only creator can appoint/cancel admins
  - only creator can transfer creator ownership
  - admins cannot remove other admins
  - only family creator can delete their own family
  - system admin can view/edit/delete any backend-persisted family/user/reset data
  - deleting a family relies on SQLite `ON DELETE CASCADE` to remove its users and password reset requests.

## Role Permissions And Navigation

- Permission source: `rolePermissions` and `navItems` in `frontend/app.js`.
- Default landing page:

## Current UI And Feature State

### Entry/Login Page

- Entry page was redesigned:
  - left side contains introduction, roles, flow, capability chips, and ledger preview
  - right side is interaction only: register, join family, login, forgot password
- Login password field has a visible-password eye toggle.
- Login checks `ADMIN_` specially and calls `/api/admin/login`; if admin ID is changed through env vars, normal-login 401 falls back to trying `/api/admin/login`.

### System Admin Backend

- System admin is independent from household `admin`.
- Frontend state stores the active system admin in `state.currentAdmin`; ordinary users still use `state.currentUserId`.
- `renderAdminData()` renders the `后台数据` page with:
  - summary cards for family/user/pending counts
  - family table: edit family name, transfer creator, delete family
  - user table: edit nickname, family role, adult/child type, household role, status, reset password
  - password reset table: reset or reject requests across all families
- System admin theme is `theme-systemAdmin` in `frontend/styles.css`, dark neutral sidebar with blue/gold accents.
- System admin backend data includes only SQLite-persisted tables: families, users, password reset requests. It does not manage localStorage transactions/budgets/tasks.

### Member Management

- Direct add member UI was removed.
- Creator/admin member page now focuses on management:
  - approve/reject pending join requests
  - reset password requests
  - active member actions
- Active member actions:
  - creator can appoint/cancel admins
  - creator can transfer creator ownership to another adult
  - creator/admin can remove allowed members
  - creator can `注销家庭`; this deletes the family and all its backend members/reset requests, then logs the creator out
- Ordinary members use readonly `家庭成员`.

### Adult Record Transactions

- Adult `记录收支` page supports:
  - expense/income/transfer/investment/repayment/reward
  - recurring weekly/monthly transactions
  - owner/member selection
  - free text notes
  - linked activity via `关联活动`
- Creator/admin can edit existing transactions:
  - list shows `修改` button only for creator/admin
  - edit mode reuses the main transaction form
  - saving updates the existing transaction and refreshes stats
  - recurring transaction edits modify only the current bill record, not the recurring template
- `关联活动` is a text input with dropdown suggestions:
  - recent transaction activities
  - event names/categories from budgets where `type === "活动"`
  - recurring bill activity names
  - fallback `日常`
- Adult main category suggestions are linked to bill type:
  - `收入`: 工资、奖金、兼职收入、经营收入、利息收入、租金收入、报销、退款、红包礼金、其他收入
  - `支出`: 餐饮、交通、住宿、教育、医疗、日用品、娱乐、人情、投资理财、零花钱发放、文具
  - `转账`: 账户互转、家庭内部调拨、零花钱发放、储蓄转入、信用卡还款、备用金调整
  - `投资`: 基金、股票、定投、银行理财、投资收益、资产配置、养老金账户
  - `还款`: 房贷、车贷、信用卡还款、借款归还、花呗/分期、助学贷款
  - `奖励`: 任务奖励、学习奖励、家务奖励、节日奖励、成长激励、其他奖励

### Child Modules

- Child record page no longer uses the old “内循环/外循环” wording in the UI.
- Child record page now uses:
  - `账单口径`
  - `真实消费`
  - `零花钱/储蓄变动`
- Child transaction save maps these to internal transaction cycle values:
  - `真实消费`
  - `个人变动`
- Child can record own income/expense/transfer/reward.
- Child `查看收支` only shows transactions owned by the current child.
- Child tasks:
  - child marks task complete
  - guardian reviews in `零花钱管理`
  - approved task creates child-owned `奖励` transaction
- Child requests:
  - child creates pending spending request
  - guardian approves/rejects
  - approved request creates child-owned `真实消费` expense and counts toward family stats
- Child savings goals are created and completed by the child; guardians can view them.

### Stats

- Statistics were changed from mostly expense-oriented to income/expense/balance oriented.
- `statsPanels()` is the shared stats workbench used by:
  - personal stats
  - family stats
  - child statement stats
- Current stats workbench shows:
  - chart view switcher: overview, trend, structure, ranking
  - summary cards with income, expense, net balance, savings rate / average expense
  - period-over-period trend badges for month, week, and recent 30-day views
  - rule-based insight panel for negative balance, low savings rate, concentrated expenses, largest expense, and risky budgets
  - income / expense / balance comparison
  - trend charts: balance line/area chart, income-expense stacked bars, income/expense vertical bars
  - structure charts: donut charts and treemap-style proportion charts
  - by time/category/activity/tag depending on allowed dimension
  - expense structure
  - income structure
  - expense ranking
  - income ranking
  - income/expense ranking
  - family manager view also shows member income/expense
- Ordinary family stats:
  - no category/tag/member detail
  - only time and activity public views

### Budget Management

- Budget management is linked to transactions now.
- Budget cards no longer trust only the old static `budget.used`; usage is computed from records in `budgetUsage()`.
- Budget matching rules:
  - activity budget: matches family-counted expense transactions whose `event` equals the budget `name` or `category`
  - category/monthly/weekly budgets: match family-counted expense transactions by `category` and period
  - budget period is set by `currentBudgetPeriod(type)` when creating new budgets
- Creator/admin can cancel budgets:
  - button: `取消预算`
  - removes only the budget item
  - does not delete transactions
- New activity budgets become suggestions in `记录收支 -> 关联活动`.
- Ordinary members can view budgets but cannot create/cancel/edit.

### AI Analysis

- AI module is still local/rule-based UI, not actually calling an LLM yet.
- It has an API endpoint input for a future backend proxy.
- It exposes a public prompt in the UI.
- AI prompt is constrained to:
  - 本月总结
  - 预算提醒
  - 活动复盘
- It must not analyze investment.
- AI data source filters to current month and excludes investment-related transactions.
- Budget reminders use computed `budgetUsage()` now.

## Backend Persistence Caveat

- Backend currently persists:
  - families
  - users
  - password reset requests
- `DELETE /api/families/{id}` and `DELETE /api/admin/families/{id}` remove persisted family data and cascade to persisted users/password reset requests.
- Backend does not yet persist:
  - transactions
  - budgets
  - recurring bills
  - child tasks
  - child requests
  - saving goals
  - AI settings
- Future work likely needs API routes and SQLite tables for these if persistence is required.
